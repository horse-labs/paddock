01 The problem

Context dies between sessions.

AI-assisted work generates a tremendous amount of context — the decisions, the roadmap, the why behind every trade-off. Then the session ends, and it evaporates.

Drive and Notion go stale the moment you close the tab. Worse, the agent can't read or write them natively. So next session opens the same way every time: re-explaining everything you already decided.

Memory shouldn't live in a human's head, or a doc nobody updates. It should live where the work lives — in the repo.

02 Memory bank — and beyond

A memory bank at its core.
Governance on top.

The category is the memory bank — structured markdown kept in the repo and fed to the agent each session to restore context. Paddock is that, then keeps going, adding the engineering governance real teams actually need.

a

Evidence-linked roadmap

What's done, what's pending — and every item linked to a PR, a commit, or an ADR. No status without proof.

b

Delivery guardrails

A lightweight session ritual, a Definition of Ready / Done, and trunk-based git. Discipline without ceremony.

c

Multi-repo + refinery

Many repos in one workspace, plus a raw _knowledge/ intake that's distilled into clean structure.

d

Cross-agent

One AGENTS.md, read by Copilot, Cursor, Codex and Claude alike. Write the context once.

03 Topology

One workspace.
The agent sees everything.

Your agent runs from a single workspace root. Code repos sit beside the structured memory, beside the raw intake. Nothing is hidden in a tab the model can't reach.

Paddock acts as a refinery: it distills the unstructured _knowledge/ — PDFs, transcripts, diagrams — into specs, decisions and status.
