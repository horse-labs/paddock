# specs/

Design de features **antes** de codar — o "como" e o "porquê". Um arquivo por spec, datado:
`AAAA-MM-DD-<slug>-design.md`.

Conteúdo típico: contexto/problema, decisão de design, alternativas descartadas, contrato/API,
trade-offs, verificação. Linkado pela coluna evidência/nota do `roadmap/status.md` (não duplicar status aqui).
